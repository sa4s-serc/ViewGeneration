KNOWLEDGE BASE: Generating a “Correct View” of Software Architecture

1. Definition of a "Correct View"
A "correct view" is a representation of one or more structural aspects of an architecture that illustrates how it addresses specific stakeholder concerns.
It must:
- Be directly supported by the given sources.
- Clearly identify elements, relations, and properties.
- Follow sound documentation best practices.

2. Purpose of a View
- An architectural view represents a set of system elements and their relationships.
- Enables separation of concerns, breaking down complex architectures into comprehensible representations.

3. Core Structural Components
Each view should contain:
1. Elements – constituent parts of the system shown.
2. Relations – connections/interactions between elements.
3. Properties – important characteristics of elements and relations.

4. Categories & Styles of Views

A. Module Views/Styles
- Purpose: Show the system's units of implementation.
- Examples: Decomposition, Uses, Generalization, Layered, Aspects, Data Model.
- Elements: Units of implementation.
- Relations:
  - is-part-of
  - shares-part-of-the-same-secret-as
  - depends on the correctness of
  - gives computational work to
  - uses
  - is-a
  - crosscuts
- Properties: Responsibility, visibility of interfaces, implementation details.

B. Component-and-Connector (C&C) Views/Styles
- Purpose: Show runtime units of execution and their interactions.
- Examples: Data Flow, Call-Return, Event-Based, Repository.
- Elements: Components (runtime entities, services) and Connectors (interaction pathways).
- Relations:
  - attaches-to
  - sends-data-to
  - requests-service-from
  - publishes-to
  - subscribes-to
  - accesses-data
- Properties: Concurrency, performance, security, reliability.

C. Allocation Views/Styles
- Purpose: Show relations between software and non-software resources.
- Examples: Deployment, Install, Work Assignment.
- Elements: Software mapped to non-software entities (hardware, teams).
- Relations:
  - allocated-to
  - migrates-to
  - copy-migrates-to
  - executes-on
  - installs-on
  - assigned-to
- Properties: CPU, memory, disk capacity, safety-critical properties, fault tolerance.

5. Documenting the View
- Prefer formal (e.g., UML) or semi-formal notations for precision.
- Always explain notation with a key/legend:
  - Shapes (rectangles, icons)
  - Colors (and their meanings)
  - Connector types and directions
- Specify granularity (high, medium, low) and scope (part of system, whole system, whole + external).
- Address concerns (architecture documentation, deployment, data flow, etc.) and quality attributes (maintainability, performance, security, etc.).
- Specify static or dynamic properties.
- Identify technologies used (e.g., AWS, Kubernetes, Docker).
- Include design overlays (textual descriptions, screenshots, URLs, code snippets, configs).

6. Sound Documentation Rules
1. Write from reader’s perspective (stakeholder-driven) which can be captured from concerns and quality attributes.
2. Avoid ambiguity — be explicit and precise.

7. Real-World Observations & Practices
- Multiple views are necessary; no single view shows the entire architecture.
- Common concerns: general architecture documentation, deployment, control flow.
- Granularity: mostly high/medium level (layers, subsystems, key technologies).
- Common styles: client-server, layered, service-oriented, event-driven.
- Notations: 96% informal (boxes & lines); legends rarely included (8%).
- Shapes: rectangles (79%), icons (46%).
- Colors: used in 81% of views but meaning often undefined.
- Connectors: 92% explicit; mostly unidirectional (74%), some bidirectional (29%).
- Technologies mentioned in 35% of views (AWS, Kubernetes, Docker common).
