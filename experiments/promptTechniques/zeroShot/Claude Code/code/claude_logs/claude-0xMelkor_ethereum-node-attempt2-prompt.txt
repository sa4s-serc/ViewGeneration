## Ethereum Node Deployment on GKE: Repository Analysis

*   **Endpoint Exposure via Reverse Proxy:** Utilizes Nginx as a reverse proxy to expose Geth RPC (HTTP and WebSocket) and InfluxDB endpoints for external access. This allows secure and controlled interaction with the node.
*   **CI/CD Pipeline:** Leverages Google Cloud Build for continuous integration and continuous deployment (CI/CD), automating the deployment process.

**Architecture and Design:**

*   **Microservices Architecture:** Decomposes the Ethereum node into Geth and Lighthouse components, deployed as separate microservices. This facilitates independent scaling, updates, and fault isolation.
*   **Kubernetes Orchestration:** Employs Kubernetes to orchestrate the deployment, scaling, and management of the Ethereum node, InfluxDB, and Nginx, providing a robust and scalable platform.
*   **Service Discovery:** Utilizes Kubernetes Services (ClusterIP and potentially NodePort for external access) for service discovery, enabling inter-component communication within the cluster.  InfluxDB uses a headless service for stable network identifiers, a best practice for stateful applications.
*   **Persistent Storage:** Provisioning of persistent storage for Geth, Lighthouse, and InfluxDB data directories using PersistentVolumeClaims ensures data persistence across pod restarts.

**Important Files and Their Roles:**

*   **`cloudbuild.yaml`:** The cornerstone of automated deployments. This file defines the Cloud Build configuration, orchestrating steps such as secret substitution, Kubernetes manifest application, and infrastructure setup. It uses `envsubst` for secure secret injection from Google Cloud Secret Manager.

*   **Secrets Management:** Employs Google Cloud Secret Manager for securely storing and managing secrets.  `envsubst` then injects these secrets as environment variables into Kubernetes pods, preventing hardcoding in manifests.
