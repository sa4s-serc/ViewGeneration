## FluxSurfer: Analysis Report

**Summary:**

The "FluxSurfer" repository provides a C++ library for simulating the dynamics of finite-state quantum systems. It offers analytical and numerical solvers for Ordinary Differential Equations (ODEs), along with a Monte Carlo simulation method, facilitating parallel execution and structured data storage. The core concept involves representing a quantum system as a graph, where nodes represent states and edges represent transitions between states, with edge weights corresponding to transition probabilities.

**Key Functionalities and Components:**

*   **QuantumSystem Class:** This is the central class, representing a quantum system. It stores the graph structure (states and transitions), manages occupation data, and handles data saving to `.graphml` files.
*   **Solvers:** The library provides multiple solvers for the simulation:
    *   **ODE Solvers:** Numerical solvers for the master equation (a first-order ODE), including adaptive and fixed step size solvers based on Richardson extrapolation and the Euler method. These solvers utilize the abstract base classes `SingleStepScheme` and `Solver`
    *   **MonteCarloWanderer:** A Monte Carlo algorithm that directly explores the graph to simulate time evolution.
*   **Experiment Class:** This class manages parallel processing, metadata storage, and coordinates the simulation of multiple time evolutions of a quantum system. It defines how the simulation should be carried out.
*   **Parallelization:** Implemented via multi-threading for parallel execution of experiment measurements, utilizing a shared queue (`WorkSource`) and a mutex for thread synchronization.
*   **Data Storage:** Simulation data is saved in a structured manner:
    *   `.graphml` files: Contain the graph structure, node occupations, and edge weights (using XML format).
    *   `METADATA.json`: Human-readable metadata (descriptions, motivations, constant parameters).
    *   `METADATA.csv`: Table containing parameters that varied during different measurements within the experiment.
*   **Utilities:** Additional utilities for binary number representation and XML file parsing are available.

**Important Files and Their Roles:**

*   `compile.txt`: Compilation command for g++, linking to libraries like Armadillo, pthread, and jsoncpp.
*   `LICENSE`: MIT License.
*   `README.md`: Library overview, structure, and usage.
*   `Doxyfile`: Configuration file for Doxygen API documentation.
*   `./tools/fileUtility.py`: Python script for parsing the XML-based `.graphml` data files.
*   `./tools/quantities.py`: Python script for calculating physical quantities (e.g., total particle number, level occupation).
*   `./tools/plotChargeCurve.py`: Python script for plotting charge curves.
*   `./tools/menu.py`: Generates Terminal Textmenus.
*   `./tools/binaryNumberUtility.py`: Provides functions to work with binary representations of state IDs.

**Architectural Insights:**

*   **Graph-Based Representation:** The core architectural decision is the graph-based representation of quantum systems, facilitating the use of graph algorithms.
*   **Modularity:** The library exhibits a modular design with clear separation of concerns. `QuantumSystem` models the physics, `Solvers` handle time evolution, `Experiment` manages parallel execution and data storage, and `tools` provide post-processing utilities.
*   **Extensibility:** The use of abstract base classes like `Solver` and `SingleStepScheme` allows for easy addition of new solvers and integration schemes.
*   **Parallel Processing Design:** Employs a producer-consumer pattern for parallel processing using a shared queue and mutex.

**Notable Design Patterns:**

*   **Abstract Base Class:** The `Solver` and `SingleStepScheme` classes promote polymorphism and extensibility.
*   **Template Method:** The `SingleStepODESolver`'s `solve()` method utilizes the `SingleStepScheme`, implementing the template method pattern.
*   **Strategy Pattern:** The `SingleStepODESolver` uses the `SingleStepScheme` as the algorithm to use at each step. The difference with template method is, that the algorithm can be chosen at runtime.
*   **Producer-Consumer:** Parallel processing uses a producer-consumer pattern.

**Points for Further Consideration:**

*   **Error Handling:**  Minimal explicit error handling is present. Robust error checking and exception handling would improve reliability.
*   **Memory Management:** Dynamic memory allocation and deletion of `QuantumSystem` and `Solver` objects (as noted in `parallel_processing_code.cpp`) requires careful management to avoid memory leaks, especially in the multithreaded environment.
*   **Potential for Condition Variables:**  The thread synchronization could benefit from using condition variables instead of just using a mutex and repeatedly checking shared variables to reduce CPU usage.
