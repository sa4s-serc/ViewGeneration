ARCHITECTURAL SUMMARY:
## Indoor Map with Heatmap Application Analysis

This report analyzes an iOS application designed for displaying indoor maps with heatmap overlays, integrated with a Node.js backend deployed on Cloud Foundry. The application allows users to view floorplans and visualize event density through a heatmap, retrieving data from a backend server.

**Overview:**

The system employs a client-server architecture. The iOS app fetches map data (PDF floorplan) and heatmap data (trigger events) from the backend via RESTful API calls. The backend, built with Node.js, manages data persistence in a MongoDB database, including map data, beacon locations, booth locations, and event triggers, and generates the PDF floorplan.

**Key Functionalities:**

*   **Indoor Map Display:** Utilizes Apple's MapKit to render floorplans loaded as PDF documents from the backend.
*   **Heatmap Overlay:** Generates a heatmap visualization on top of the floorplan, based on trigger events retrieved from the backend.
*   **Data Fetching:** Employs RESTful API calls to retrieve map and event data from the Cloud Foundry-deployed backend.
*   **Coordinate Conversion:** Translates between PDF coordinates (used in the floorplan) and geographic coordinates (latitude/longitude) required by MapKit.
*   **Map Camera Management:** Restricts map scrolling and zooming to remain within the floorplan boundaries.
*   **Event Triggering:** Includes utilities (particularly in `random.js`) to generate and send simulated events to the backend to populate the heatmap, though the iOS app can trigger events via POST requests.
*   **Backend Data Storage:** Uses MongoDB to persist beacon, booth, and event data.
*   **PDF Generation:** Backend component responsible for creating the PDF floorplan based on data stored in the database.

**Important Files and Their Roles:**

*   **iOS Application (heatmap/):**
    *   `ViewController.swift`: Main view controller, responsible for map initialization, data fetching, overlay rendering, user interaction, and location updates.
    *   `FloorplanOverlay.swift`: Defines the `FloorplanOverlay` class (an `MKOverlay`), which handles loading the PDF floorplan, coordinate conversion, and map drawing.
    *   `FloorplanOverlayRenderer.swift`: Renders the `FloorplanOverlay` on the `MKMapView`, specifically drawing the PDF floorplan and diagnostic visuals.
    *   `CoordinateConverter.swift`: Manages the conversion between PDF coordinates and geographic coordinates, using `GeoAnchor` and `GeoAnchorPair` structs.
    *   `VisibleMapRegionDelegate.swift`: Manages map camera behavior, preventing excessive scrolling/zooming.
    *   `CustomPolygons.swift`: Defines custom polygon and circle overlays used in heatmap rendering.
    *   `Utilities.swift`: Provides extension methods and utility functions for the iOS app.
*   **Backend Application (heatmap-backend/):**
    *   `app.js`: Main application file, setting up the Express server, connecting to MongoDB, and defining API routes.
    *   `routes/*.js`: Defines API endpoints for managing beacons, booths, events, SVGs, pages, and trigger events.
    *   `models/*.js`: Mongoose models defining the schema for data stored in MongoDB (e.g., `beacon.js`, `booth.js`, `event.js`, `page.js`, `trigger.js`).
    *   `manifest.yml`: Cloud Foundry deployment manifest, specifying application name, memory allocation, and bound services (MongoDB).
    *   `real-data.js`: Used to insert initial beacon and booth data into MongoDB.
    *   `random.js`: Generates random data and posts it to the backend to simulate events for the heatmap.
*   **Other:**
    *   `./LICENSE`: Contains the Apache 2.0 License.
    *   `./README.md`: Contains setup and execution instructions.

**Architectural Insights:**

*   **Client-Server Architecture:** iOS client communicating with a backend server.
*   **Potential for Microservices:** Backend's modular structure (routes and models) allows for easier future development into microservices, separating concerns like beacon or event management.
*   **Loose Coupling:** The iOS application interacts with the backend through a well-defined API, decoupling them and allowing for independent evolution.
*   **Cloud Foundry Deployment:** The backend is designed for deployment on Cloud Foundry.

**Design Patterns:**

*   **Model-View-Controller (MVC):** iOS app follows MVC with `ViewController.swift` as controller.
*   **Data Access Object (DAO):** Mongoose models in the backend act as DAOs, abstracting data access from MongoDB.
*   **Delegate:** `VisibleMapRegionDelegate` handles map camera events.
