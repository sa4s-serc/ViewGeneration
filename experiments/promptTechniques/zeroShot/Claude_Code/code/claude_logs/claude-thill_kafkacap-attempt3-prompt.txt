## KafkaCap Repository Analysis Report

This report summarizes the key functionalities, architecture, and design patterns found in the KafkaCap repository. KafkaCap provides a framework for reliably capturing messages to Kafka using redundant capture devices and a deduplicator consumer group. The core goal is to ensure messages are uniquely written to a single Kafka topic, even in the presence of failures or redundancy.

**Core Functionality:**

KafkaCap achieves reliable message capture through three primary mechanisms:

1.  **Redundant Capture Devices:** Messages are captured from various sources (Aeron, Multicast, Websocket) via multiple capture devices. Each device writes to a separate Kafka topic.
2.  **Deduplicator:** A Kafka Consumer Group consumes from all capture topics. It deduplicates messages, ensuring only unique messages are written to the outbound Kafka topic.
3.  **Buffered Publisher:** Messages are buffered before being published to Kafka. This mitigates back-pressure and provides resilience during Kafka outages.

**Key Components:**

*   **CaptureDevice (Abstract Class):** Defines the interface for capturing messages from various sources. Concrete implementations exist for Aeron (`AeronCaptureDevice`), Multicast (`MulticastCaptureDevice`), and Websocket (`WebsocketCaptureDevice`). The `AeronDedupStrategy` builds on the MultiProducerDedupStrategy to use Aeron's stream ID.
*   **Deduplicator:** The central component responsible for deduplicating messages. The main class is `Deduplicator.java` in the `kafkacap-core` module.
*   **DedupStrategy (Interface):** Defines the deduplication logic. Implementations include `SequencedDedupStrategy` (for sequenced streams) and `MultiProducerDedupStrategy` (for streams with multiple logical producers).
*   **CaptureQueue (Interface):** Manages buffering of inbound messages. Available implementations include `MemoryCaptureQueue` (in-memory) and `ChronicleCaptureQueue` (disk-based, using Chronicle-Queue).
*   **RecordPopulator:** Creates Kafka records from the captured messages.
*   **RecoveryService:** Recovers deduplication state upon restart by reading the last committed offsets from the outbound Kafka topic.

**Architectural Overview:**

KafkaCap follows a modular, pluggable, and configuration-driven architecture. Its design emphasizes fault tolerance and relies heavily on Kafka's features.

*   **Modular Design:** The repository is organized into modules (core, aeron, multicast, websocket), enabling easy extension with new capture devices.
*   **Pluggable Components:** Interfaces like `CaptureDevice`, `DedupStrategy`, and `CaptureQueue` allow users to customize the system with their own implementations.
*   **Fault Tolerance:** Redundant capture devices and Kafka Consumer Groups contribute to high availability.
*   **Kafka-Centric:** Leverages Kafka's topics, partitions, and consumer groups for message storage, deduplication, and fault tolerance.
*   **Configuration-Driven:** YAML files configure both capture devices and the deduplicator, promoting flexibility and ease of deployment. Example Deduplicator configuration can be found in `dedup_demo.yaml`. The general architecture is documented in `kafkacap_architecture.drawio`

**Notable Design Patterns:**

*   **Strategy:** The `DedupStrategy` interface exemplifies the Strategy pattern, allowing the deduplication logic to be easily swapped.
*   **Template Method:** The `CaptureDevice` abstract class uses the Template Method pattern, defining the skeleton of an algorithm while allowing subclasses to implement specific steps.
*   **Factory:** The `RecordCacheFactory` interface demonstrates the Factory pattern, abstracting the creation of RecordCache instances.
*   **Builder:** The `BufferedPublisherBuilder` and `DeduplicatorBuilder` simplify the creation of complex objects with many optional parameters.

**Conclusion:**

KafkaCap provides a robust and flexible framework for capturing and deduplicating messages into Kafka. Its modular design, pluggable components, and reliance on proven design patterns contribute to its maintainability and extensibility. The use of redundant capture devices and Kafka's features ensure high availability and fault tolerance.
