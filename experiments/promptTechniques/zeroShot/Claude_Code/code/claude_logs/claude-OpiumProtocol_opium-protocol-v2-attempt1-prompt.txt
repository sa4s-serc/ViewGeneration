**Architectural Insights:**

*   **Proxy-Based Upgradeability:** The architecture is designed to be upgradeable, utilizing proxy contracts (e.g., `Registry_Proxy`, `Core_Proxy`) and implementation contracts (e.g., `Registry_Implementation`, `Core_Implementation`). This allows the underlying contract logic to be updated without changing the address used by users and other contracts.
*   **Centralized Registry:** The Registry contract acts as a central point for configuration and addresses of other key contracts, simplifying management and allowing dynamic updates.
*   **Modular Design:** The protocol is broken down into distinct, specialized contracts and libraries with well-defined responsibilities, promoting reusability and maintainability.
*   **Role-Based Access Control:** The Registry utilizes a role-based access control system (using OpenZeppelin's `AccessControl`) to manage permissions for different functions, such as pausing the protocol or updating protocol parameters. *   **Dependency Injection:** Core components are injected with other contract addresses (e.g., the `Core` contract receives the `Registry` address during initialization), enabling loose coupling and modularity.
*   **Clear Separation of Concerns:** The various aggregators help separate concerns, such as data fetching from derivative logic.

*   **Registry Pattern:** Utilized for contract discovery and parameter management (similar to a service locator).
*   **Factory Pattern:** The `OpiumProxyFactory` is responsible for creating instances of position token contracts.
*   **Access Control:** The Registry contract uses the Access Control design pattern through OpenZeppelin's AccessControl library.
*   **Data Caching:** The `SyntheticAggregator` caches derivative data to prevent tampering.
