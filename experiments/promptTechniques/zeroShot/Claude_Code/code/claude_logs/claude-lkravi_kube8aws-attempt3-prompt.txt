**Overview:**

This Terraform configuration automates the deployment of a Kubernetes cluster on AWS using Kubeadm and Ansible. It embodies Infrastructure as Code (IaC) principles, defining and provisioning the necessary AWS resources and subsequently configuring the cluster in an automated and consistent manner. The architecture leverages a Bastion host for secure access and utilizes a Network Load Balancer (NLB) to ensure a highly available Kubernetes API server endpoint. `cri-dockerd` enables Docker to be used as the cluster's container runtime.

**Key Functionalities:**

*   **Automated Infrastructure Provisioning (Terraform):**  Provisions the AWS infrastructure: VPC, subnets, security groups, EC2 instances (Bastion, Masters, Workers), and NLB.  A Terraform module (`terraform-aws-modules/vpc/aws`) simplifies VPC configuration, promoting reusability.
*   **Automated Kubernetes Cluster Setup (Ansible):** Configures the Kubernetes cluster using Kubeadm, installing necessary packages, and setting up networking. This includes initializing the control plane and joining worker nodes.
*   **Secure Access & Control:**  The Bastion host serves as a secure jump box for SSH access to the cluster nodes located within private subnets. It also functions as the Ansible control node, executing the playbooks.
*   **Highly Available API Server:** An NLB provides a stable and highly available endpoint for the Kubernetes API server, crucial for a resilient control plane.

**Component Breakdown & File Roles:**

*   **`vpc.tf`:** Defines the Virtual Private Cloud (VPC) leveraging the `terraform-aws-modules/vpc/aws` module.
*   **`servers.tf`:** Defines EC2 instances for Bastion, master, and worker nodes, including initial setup via user data for the Bastion.
*   **`load_balance.tf`:** Creates and configures the NLB with target groups and listeners for the Kubernetes API server, using IP-based targets.
*   **`security.tf`:** Defines security groups controlling access (SSH) and communication between Kubernetes nodes and components.
*   **`keys.tf`:** Generates an SSH key pair, storing the private key locally and creating an AWS key pair resource.
*   **`ansible_helper.tf`:** Generates the Ansible inventory from the `templates/inventry.tftpl` template, copies Ansible playbooks to the Bastion, and executes them. It includes a `time_sleep` resource as a basic retry/wait mechanism to ensure the Bastion is ready before Ansible execution.
*   **`templates/inventry.tftpl`:** Template for dynamically generating the Ansible inventory, listing master and worker node hostnames and IP addresses.
*   **`ansible/play.yml`:** The main Ansible playbook orchestrating the Kubernetes cluster configuration. It handles Docker installation, Kubernetes component installation (kubelet, kubeadm, kubectl), Kubeadm initialization and joining of nodes, and configuring `cri-dockerd`.
*   **`ansible/ansible_vars_file.yml`:** Contains variables passed to Ansible, including the NLB's DNS name.
*   **`outputs.tf`:** Defines output variables, notably the Bastion host's public IP address.
*   **`providers.tf`:** Configures the AWS provider, specifying the AWS region.
*   **`vars.tf`:** Defines variables used throughout the configuration, such as AWS region, AMI ID, VPC CIDR, subnet ranges, and instance types.
*   **`README.md`:** Project overview, usage instructions, prerequisites, architecture diagram, and notes on known issues.

**Design Patterns:**

*   **Modular Design:** The use of Terraform modules, especially for the VPC, promotes code reusability and simplifies configuration.
*   **Templating:** Dynamic generation of the Ansible inventory from a template ensures accurate configuration based on the provisioned infrastructure.
*   **Remote Execution:** Ansible playbooks are executed remotely on the Bastion host using Terraform's `remote-exec` provisioner.
