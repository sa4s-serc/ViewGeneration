ARCHITECTURAL SUMMARY:
## OpenShift Configuration Anomaly Detection (CAD) System Analysis Report

This report summarizes the key functionalities, components, architecture, and design patterns of the `openshift_configuration-anomaly-detection` repository. The system aims to automate the investigation and mitigation of cluster anomalies in OpenShift, triggered by PagerDuty alerts, thereby reducing manual SRE effort. The system consists of a Tekton interceptor and the `cadctl` command-line tool, which work together to detect and address anomalies.

**Key Functionalities and Components:**

*   **Automated Alert Investigation & Anomaly Detection:** The core function is to automatically investigate alerts received from PagerDuty, detecting anomalies within OpenShift clusters.
*   **`cadctl` Command-Line Tool:** This CLI handles alert remediation and pre-investigations, triggered by PagerDuty webhooks. The `cadctl investigate` subcommand is the main entry point for handling alerts.
*   **Tekton Interceptor:** Filters incoming PagerDuty webhooks, validating signatures and determining if `cadctl` has an implemented handler function for the specific alert.
*   **Tekton PipelineRuns:** When an alert is handled, a Tekton PipelineRun is initiated, executing `cadctl` to perform the investigation.
*   **Limited Support Reasons:** Automated setting of limited support reasons based on detected anomalies.
*   **Ready Made Investigations:** The system includes a number of ready made investigations based on an extensible framework. Some examples of investigations include: `chgm`, `apierrorbudgetburn`, `insightsoperatordown`, `ccam`, `upgradeconfigsyncfailureover4hr`, `cpd`, `clustermonitoringerrorbudgetburn` and `machinehealthcheckunterminatedshortcircuitsre`.

**Integration with External Systems:**

The system integrates with several external systems:

*   **PagerDuty:** Receives alerts and updates incident information.
*   **OCM (OpenShift Cluster Manager):** Retrieves cluster information and interacts with support reasons/service logs.
*   **AWS:** Used for cluster login, retrieval of instance information, and interaction with AWS CloudTrail. `osd-network-verifier` is used via the `pkg/networkverifier` package to validate network configurations.
*   **Kubernetes:** Interacts with Kubernetes to fetch and inspect cluster resources.

**Architectural Insights:**

*   **Event-Driven Architecture:** Utilizes PagerDuty webhooks to trigger automated investigations.
*   **Microservice-like Design:** Deploys the Tekton interceptor as a service with isolated binaries/images.
*   **Plugin-like Structure:** The use of the `Investigation` interface (`pkg/investigations/investigation/investigation.go`) promotes a plugin-like architecture. New investigations can be added by implementing the interface and providing the necessary metadata.
*   **Dependency Injection:** The `investigation.Resources` struct serves as a container for dependencies required by the investigations, enabling easier testing and dependency management. The `Resources` struct acts as a container for dependencies (clients, data), which are injected into the `Run` method of each investigation.
*   **Configuration-Driven:** Uses `update.cfg` to allow to select which features to include.
*   **Error Handling:** Demonstrates careful error handling, often wrapping errors with contextual information and using `errors.Join` to accumulate errors.
*   **Logging and Observability:** Leverages the `go.uber.org/zap` logging library. Each investigation utilizes a `NoteWriter` for consistent formatting of results appended to PagerDuty.
*   **Backplane Integration:** Employs backplane-cli for managing cluster RBAC for the investigations using dynamic Service Accounts, enabling least privilege access.

**Notable Design Patterns:**

*   **Strategy Pattern:** The `investigations` package implements a strategy pattern to select the appropriate Investigation based on the alert title.
*   **Service Locator Pattern:** The `investigation.Resources` struct can be seen as a service locator, providing access to various services needed during an investigation.
*   **Interface-based Programming:** The `Investigation` interface defines a contract that all investigation implementations must adhere to.

This architecture emphasizes modularity, testability, and maintainability, facilitating the addition of new investigations and adaptation to evolving cluster environments. The framework allows for modular addition of new investigations and provides dependency management for ease of testing.
