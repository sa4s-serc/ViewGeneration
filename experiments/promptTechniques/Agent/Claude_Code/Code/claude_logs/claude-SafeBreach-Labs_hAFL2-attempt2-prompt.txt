## hAFL2 Repository Analysis Report

This report summarizes the analysis of a GitHub repository focused on hypervisor fuzzing using hAFL2, a kAFL-based fuzzer. hAFL2 extends the original kAFL to support nested virtualization and is designed for fuzzing hypervisor components, specifically Virtualization Service Providers (VSPs) within Hyper-V. The analysis covers the core fuzzer implementation, kernel drivers, QEMU modifications, and Redqueen integration crucial for its operation.

**1. Core Fuzzing Implementation (kAFL):**

The heart of the framework is the kAFL fuzzer, which leverages KVM and Intel PT (Processor Trace) for coverage feedback and leverages KVM control over nested VMs to communicate with harness. The GUI (`kafl_gui.py`) provides a text-based interface to monitor fuzzing statistics. `kafl_fuzz.py` launches the fuzzer, employing a master-slave architecture for parallel execution. The master (`master.py`) manages the fuzzing queue and task distribution, while slave processes (`slave.py`) execute the target in QEMU/KVM, mutate inputs, and report results. Mutation techniques include bit flipping, arithmetic mutations, and leveraging "interesting values." `kafl_user_prepare.py` packages Linux userspace binaries for fuzzing. Tools like `kafl_cov.py` (coverage reporting), `kafl_debug.py` (targeted input execution), and `kafl_plot.py` (visualization of input discovery) support the fuzzing process.

**2. Hyper-V Fuzzing Scenario & Nested Virtualization:**

hAFL2's goal is to fuzz Hyper-V VSPs. A nested virtualization environment is created: a Linux host runs QEMU, hosting a root partition Windows VM, which in turn hosts a child partition Windows VM. The fuzzer targets the root partition of the L2 (nested) Hyper-V VM. The child partition runs the harness, sending payloads to VSPs running in the root partition. Nested virtualization is a core requirement, enabled by modifications to KVM.

**3. Communication & Crash Monitoring:**

Custom Windows kernel drivers are crucial for communication and crash handling. `CPHarness.sys`, running in the child partition, sends fuzzing inputs (RNDIS packets) to the VMSwitch (vmswitch.sys) in the root partition and receives completion status. `CrashMonitoringDriver.sys`, in the root partition, monitors crashes and reports information (registers, stack traces) back to the fuzzer via hypercalls. Hypercalls facilitate guest-host communication. KVM is modified to ensure that kAFL-originated hypercalls aren't reflected to the L1 hypervisor.

**4. Code Coverage with Intel PT:**

Intel PT provides detailed code coverage. KVM is modified to enable PT exclusively for the root partition VM, allowing targeted coverage feedback. Modifications to KVM's VM-Entry and VM-Exit handlers enable and disable Intel PT. `EXIT_REASON_TOPA_FULL` indicates when Intel PT buffers are full. Monitor commands (defined in `softmmu/hmp-commands-pt.h`) control PT settings interactively.

**5. Redqueen Instrumentation (for feedback driven fuzzing):**

Redqueen, a feedback-driven fuzzing technique, is integrated to enhance vulnerability discovery. The code includes mechanisms for hooking into program execution (`handle_hook`), disassembling instructions using the Capstone engine, and extracting register values and memory access details (especially in symbolic execution mode - SE).  Breakpoint handling, single-stepping, and tracing based on blacklists and whitelists are implemented. The system supports different instrumentation modes (light, SE, whitelist). The collected SE data is intended for processing by an external symbolic execution engine.
